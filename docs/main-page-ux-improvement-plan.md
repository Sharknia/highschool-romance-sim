# Main Page UX Improvement Plan

작성일: 2026-05-20

## 결론

현재 `apps/web`의 메인페이지는 Alpha 기능의 진입점들을 한 화면에 올려두었지만, 제작자가 실제로 따라야 할 순서와 현재 상태를 설명하지 못한다. 따라서 "기능 구현 완료"와 "사용 가능한 제작툴 UX 완료"를 분리해서 판단해야 한다.

이 문서의 범위는 Beta 반복 제작 확장이 아니다. 범위는 Alpha happy path를 사용자가 막힘 없이 완주하게 만드는 메인페이지 사용성 보장이다.

Alpha UX 완료 기준은 사용자가 JSON을 직접 이해하지 않아도 다음 흐름을 완주하는 것이다.

1. 프로젝트를 만들거나 연다.
2. 히로인을 만들고 저장한다.
3. 히로인을 기준으로 프로젝트를 생성한다.
4. 자연어 이벤트를 입력한다.
5. 패치 제안, 검증 결과, 변경 내용을 이해한다.
6. 승인 후 CG를 생성한다.
7. 플레이 프리뷰로 확인한다.
8. export 결과를 확인한다.

## 확인한 증거

- 코드 기준: `apps/web/src/client/pages/WorkspacePage.tsx`가 히로인 라이브러리, 자연어 이벤트, 플레이 프리뷰, 프로젝트 JSON, 결과 로그를 한 컴포넌트와 한 화면에 배치한다.
- 코드 기준: `apps/web/src/client/styles.css`는 데스크톱에서 좌측 사이드와 우측 메인 컬럼을 제공하지만, 사용자 여정이나 작업 모드 전환보다는 패널 나열에 가깝다.
- 브라우저 실제 실행: `npm run dev -w @vn-maker/web` 후 `http://127.0.0.1:5173/`에서 워크스페이스가 렌더링되고 콘솔 오류는 보이지 않았다.
- 브라우저 실제 실행: 워크스페이스 첫 화면에 버튼 18개와 textarea 5개가 노출된다.
- 목 테스트: 모바일 폭 확인은 Codex 세션 응답을 목으로 고정해 워크스페이스 렌더와 정보 순서를 확인했다.

## 현재 UX 문제

### 1. 첫 행동이 보이지 않는다

사용자는 "이제 무엇을 눌러야 하는가"를 화면에서 바로 알 수 없다. `목록`, `저장`, `포트레이트 생성`, `프로젝트 생성`, `패치 제안`, `CG 생성`, `처음부터`, `현재 씬`, `Export`, `샘플`, `열기`, `검증`, `HTML`이 거의 같은 무게로 노출된다.

개선 기준:

- 현재 상태별 다음 행동은 화면에서 하나의 primary CTA로 보여야 한다.
- disabled 버튼은 왜 비활성화됐는지 설명해야 한다.
- 사용자가 Alpha happy path를 모르면 화면이 순서를 알려줘야 한다.

### 2. 정보 구조가 제작자 중심이 아니라 개발자 중심이다

`프로젝트 JSON`과 `결과`가 주요 패널로 노출되어 있다. 이는 개발자에게는 빠른 디버깅 수단이지만, 제작자에게는 "프로젝트를 만들고 있다"가 아니라 "내부 데이터를 고치고 있다"는 인상을 준다.

개선 기준:

- JSON 편집기는 기본 제작 화면이 아니라 개발자/고급 drawer로 내려야 한다.
- 결과 `pre`는 사용자가 읽어야 하는 JSON dump가 아니라 작업 로그, 오류 요약, 생성 결과 카드로 변환해야 한다.
- 수동 JSON 편집이 필요하면 "고급 편집"으로 명확히 격리한다.

### 3. 작업 상태가 흩어져 있다

프로젝트 디렉터리, 선택 히로인, 현재 씬, pending patch, planned CG job, runtime preview 상태가 서로 다른 UI 조각에 숨어 있다. 사용자는 "현재 프로젝트가 무엇이고, 저장됐는지, 다음에 무엇을 해야 하는지"를 한 곳에서 확인할 수 없다.

개선 기준:

- 페이지 상단 또는 좌측에 `프로젝트 상태`, `현재 히로인`, `현재 루트/씬`, `남은 작업`을 요약한다.
- 상태 판단은 프론트에서 임의로 재구현하지 않고 공통 use case/API 응답 경계에서 제공한다.
- 웹앱과 CLI가 같은 core와 같은 생성 어댑터를 공유한다는 제품 기준을 유지한다.

### 4. 패널이 기능 단위로 나뉘었지만 업무 단위로 묶이지 않았다

현재 패널은 `Heroine`, `Event`, `Preview`, `Project`, `Result`이다. 실제 업무는 `시작`, `캐릭터 만들기`, `이벤트 작성`, `검토 및 승인`, `프리뷰`, `내보내기`이다. 기능 이름과 업무 순서가 달라 사용자는 화면을 해석해야 한다.

개선 기준:

- 업무 단계 기반 네비게이션을 제공한다.
- 각 단계는 완료, 진행 중, 차단 상태를 가진다.
- 같은 단계 안에서만 주요 액션을 묶고, 다른 단계의 위험한 액션은 접거나 비활성화한다.

### 5. 프리뷰가 제작의 중심 무대가 아니다

미연시 제작툴의 핵심 피드백은 "내가 만든 장면이 어떻게 보이는가"다. 현재 프리뷰는 큰 패널이지만 빈 상태에서는 상태 배너만 보이고, 패치 검토나 이미지 생성 결과와 강하게 연결되지 않는다.

개선 기준:

- 중앙 영역은 가능한 한 프리뷰와 현재 작업 결과를 보여주는 stage가 되어야 한다.
- 이벤트 작성과 패치 검토는 프리뷰 결과와 연결되어야 한다.
- CG 생성 후 결과 이미지는 독립 미리보기가 아니라 해당 씬/에셋 맥락에 연결되어야 한다.

### 6. 모바일에서는 모든 것이 길게 쌓인다

현재 모바일 폭에서는 같은 패널들이 세로로 쌓이며, 사용자는 주요 CTA에 도달하기 전에 다수의 입력과 버튼을 지나야 한다.

개선 기준:

- 모바일 첫 화면에는 현재 작업명, 상태, primary CTA, 다음 단계만 먼저 보여야 한다.
- 세부 입력은 단계 내부로 접거나 탭으로 분리한다.
- JSON/debug 영역은 모바일 기본 흐름에서 숨긴다.

## 목표 정보 구조

### Top bar

- 앱 이름
- 현재 프로젝트 이름과 저장 상태
- Codex OAuth 연결 상태
- 전역 명령: 상태 갱신, 설정, 로그아웃

상단은 전역 상태만 담당한다. 개별 작업 액션을 많이 올리지 않는다.

### Left rail

- 제작 단계 stepper
- 프로젝트 열기/생성
- 히로인 라이브러리 요약
- 현재 루트/씬 요약

Left rail은 "나는 어디에 있고 다음 단계가 무엇인가"를 담당한다.

### Center workbench

모드별 작업 화면:

- 시작: 프로젝트 시작, 샘플 생성, 기존 프로젝트 열기
- 히로인: 목록, 선택, 편집, 포트레이트 생성
- 이벤트: 자연어 입력, 삽입 위치, 패치 제안
- 검토: diff, 검증 결과, 승인/취소
- 프리뷰: 처음부터/현재 씬 플레이, 선택지 이동, CG 확인
- 내보내기: 검증, export, 결과 위치와 smoke check

Center workbench는 현재 단계의 실제 작업만 보여준다.

### Right inspector

- 현재 선택 대상의 상세 정보
- 생성 작업 상태
- 검증 오류와 해결 액션
- 작업 로그 요약

Right inspector는 맥락형 보조 영역이다. JSON dump가 아니라 사용자가 처리해야 할 정보만 보여준다.

### Developer drawer

- 프로젝트 JSON
- raw API response
- patch raw payload
- export HTML/build artifact 확인

개발자 도구는 기본 사용성 흐름에서 분리한다.

## 개발 태스크 구조

아래 태스크는 선행 관계가 있다. 순서를 무시하면 프론트가 도메인 상태를 중복 소유하거나, JSON dump를 다시 주 UI로 끌어올리는 회귀가 생긴다.

태스크는 Alpha UX completion gate로 관리한다. 검색/정렬/태그, 기본 감정 에셋 5종, 이미지 작업 보드, 패치 히스토리 같은 Beta 반복 제작 기능은 이 문서의 구현 범위가 아니다.

### UX-01. 워크플로 상태 모델과 next action 계약 정의

목표:

- 프로젝트 상태를 `empty`, `projectReady`, `heroineReady`, `patchPending`, `patchApproved`, `assetsReady`, `previewReady`, `exportReady` 같은 사용자 여정 상태로 요약한다.
- API 또는 공통 use case가 다음 행동과 차단 사유를 제공한다.
- 프론트는 상태를 표시하고 액션을 호출할 뿐, 도메인 규칙을 재판단하지 않는다.

완료 조건:

- 웹앱과 CLI가 공유 가능한 상태 요약 타입이 있다.
- 각 상태의 primary CTA, secondary CTA, disabled reason이 정의된다.
- `WorkspacePage`가 자체 fallback 판단을 늘리지 않는 구현 계획이 있다.

선행:

- Alpha happy path에 필요한 최소 상태 계약은 이 작업 안에서 먼저 정의한다.
- 공통 use case 리팩토링과 방향은 맞추되, 큰 리팩토링 완료를 기다린다는 의미가 아니다.

### UX-02. 메인페이지 정보 구조와 레이아웃 재구성

목표:

- top bar, left rail, center workbench, right inspector, developer drawer로 책임을 나눈다.
- `Heroine/Event/Preview/Project/Result` 패널 나열을 업무 단계 기반 화면으로 바꾼다.

완료 조건:

- 데스크톱 첫 화면에서 primary CTA는 1개, 같은 단계의 secondary CTA는 최대 3개만 노출된다.
- JSON 편집기와 raw result는 기본 화면에서 보이지 않는다.
- 각 영역의 책임이 컴포넌트 이름과 props로 드러난다.

선행:

- UX-01 이후.

### UX-03. 시작/빈 상태와 프로젝트 선택 흐름 개선

목표:

- 처음 온 사용자가 `샘플`, `새 프로젝트`, `기존 프로젝트 열기` 중 무엇을 해야 하는지 알 수 있게 한다.
- 프로젝트 디렉터리 입력을 원시 path field가 아니라 프로젝트 선택/생성 흐름으로 감싼다.

완료 조건:

- 빈 프로젝트 상태에서 다음 행동이 하나로 보인다.
- 프로젝트가 열리면 저장 위치, 프로젝트 이름, 검증 상태가 상단 요약에 반영된다.
- API 오류, 빈 응답, 비JSON 응답은 사용자용 문장으로 표시된다.

선행:

- UX-02 이후.

### UX-04. 히로인 라이브러리 UX를 목록, 편집, 에셋으로 분리

목표:

- 히로인 목록과 편집 폼을 분리하고, 선택된 히로인의 저장/삭제/프로젝트 생성 가능 여부를 명확히 한다.
- 말투와 외형을 하나의 textarea에 합치지 않는다.
- 포트레이트 생성 결과가 히로인 에셋으로 연결됐는지 표시한다.
- 검색/정렬/태그, 히로인 복제, 감정 에셋 세트 생성은 Alpha 범위가 아니므로 포함하지 않는다.

완료 조건:

- 히로인 생성, 수정, 삭제, 프로젝트 생성의 상태와 결과가 각각 구분된다.
- 저장 전 변경 사항이 있으면 표시된다.
- 삭제는 프로젝트 스냅샷 영향 없음 정책을 UI에서 설명한다.

선행:

- UX-02 이후.

### UX-05. 자연어 이벤트 작성과 패치 검토 흐름 개선

목표:

- 이벤트 작성, 삽입 위치 선택, 패치 제안, diff 검토, 승인/취소를 단계형으로 묶는다.
- 승인 버튼의 활성 조건과 차단 사유를 명확히 표시한다.

완료 조건:

- 패치 제안 결과가 숫자 요약만이 아니라 추가/수정될 씬, 선택지, CG job 카드로 보인다.
- 검증 실패는 path와 raw message만 나열하지 않고 사용자가 고칠 수 있는 문장으로 표시된다.
- 승인 전 원본 유지, 승인 후 저장 완료 상태가 명확히 보인다.

선행:

- UX-02 이후.
- 기존 패치 diff/검증/승인 기능 작업과 연결한다.

### UX-06. 프리뷰와 export를 완료 검증 흐름으로 연결

목표:

- 프리뷰를 중앙 stage로 만들고, 현재 씬/처음부터 플레이가 같은 맥락에서 동작하게 한다.
- export는 단순 버튼이 아니라 검증, 산출물 위치, smoke check 결과까지 보여주는 완료 단계로 만든다.

완료 조건:

- 프리뷰 빈 상태는 왜 비었는지와 다음 행동을 보여준다.
- CG 생성 후 해당 씬에서 이미지가 보이는지 확인할 수 있다.
- export 완료 후 산출물 위치, 검증 결과, 실패 사유가 표시된다.

선행:

- UX-05 이후.
- 기존 프리뷰/export 구현 작업과 연결한다.

### UX-07. 개발자 도구 격리와 결과/오류 표준화

목표:

- 프로젝트 JSON, raw API response, raw patch, build HTML을 developer drawer로 내린다.
- 사용자용 결과 메시지, 오류 메시지, 작업 로그 포맷을 통일한다.

완료 조건:

- 기본 제작 흐름에서 raw JSON을 볼 필요가 없다.
- 개발자 drawer를 열면 기존 디버깅 정보에 접근할 수 있다.
- 빈 응답, 비JSON, 5xx 응답이 UI를 깨지 않고 재시도/확인 액션을 제공한다.

선행:

- UX-02 이후.

### UX-08. 데스크톱/모바일 반응형과 접근성 검증

목표:

- 재구성한 화면을 데스크톱과 모바일 폭에서 실제 브라우저로 검증한다.
- 버튼 텍스트, 입력 label, disabled reason, focus order를 점검한다.

완료 조건:

- 데스크톱과 모바일 Playwright 또는 agent-browser 확인 기록이 남는다.
- Alpha happy path를 브라우저에서 1회 실제 수행한다.
- 모바일에서 primary CTA가 첫 화면에 보이고 JSON/debug 영역이 기본으로 숨겨진다.

선행:

- UX-03, UX-04, UX-05, UX-06, UX-07 이후.

## 불가 판정 기준

다음 상태이면 Alpha UX 완료로 볼 수 없다.

- 사용자가 JSON textarea를 이해해야 주요 흐름을 진행할 수 있다.
- 첫 화면에 같은 위계의 CTA가 10개 이상 노출된다.
- 프로젝트가 어떤 상태인지 상단 또는 단계 요약에서 알 수 없다.
- disabled 버튼의 이유를 알 수 없다.
- 모바일에서 현재 단계와 primary CTA보다 raw form/debug 정보가 먼저 나온다.
- 패치 승인 전후의 프로젝트 저장 상태를 구분할 수 없다.
- export 버튼을 눌렀을 때 실제 산출물 확인 또는 실패 이유가 사용자용으로 표시되지 않는다.
